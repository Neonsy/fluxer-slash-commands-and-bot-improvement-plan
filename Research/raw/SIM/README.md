# Bot Platform Simulations

This directory validates the accepted QAD contracts against the verified current state. It is not a claim that the simulated behavior exists. Unless a scenario explicitly says otherwise, every state, route, worker, record, component, command, interaction, and suspension behavior below is planned behavior and an implementation gap today.

## How to read the simulations

- `Decision evidence` names exact QAD IDs and the trail/spec heading that controls the result. A simulation never supplies policy that QAD leaves open.
- `Current-state evidence` names both the exact CS section and concrete source files inspected at the research baseline. Source paths are repository-relative.
- `Expected processing` is an implementation acceptance trace: every numbered step must be independently observable or testable.
- `Conclusion` is justified from the state/effect trace. `Validated design` means the accepted decisions determine a coherent result, not that current code passes the scenario.
- `Gap` means no current implementation exists. `Decision gap` means the accepted corpus does not determine an answer; the scenario fails closed instead of inventing one.

## Cross-scenario invariants

1. The canonical domain row/head and its version are authoritative. Queue delivery, cache state, UI state, audit emission, and indexes are not transaction proofs.
2. Installation authority requires `ACTIVE` plus no suspension. Leaving `ACTIVE`, permanent deletion, suspension, generation change, authority revision change, dismissal, or expiry revokes applicable authority before cleanup.
3. Names, aliases, labels, application text, client permission snapshots, and resource presence never establish identity or authorization.
4. Retry reuses a durable operation/effect identity. It does not allocate another role, message, interaction, command identity, capability, audit effect, or delivery payload.
5. Reinstall/reinstatement/recovery may authorize only new generation/epoch activity. Old responses, component claims, confirmations, and delegated capabilities never reactivate.
6. Rollback disables new producers/discovery but leaves tolerant readers, revocation, cleanup, suspension, audit, expiry, and reconcilers operational.
7. Capability fallback, picker ordering/grouping, settings navigation, and dashboard presentation never grant authority or weaken server preflight.

## Surfaced gaps and coordinator rechecks

- **Implementation gap (universal):** none of the proposed bot-platform domains exists today. These are executable acceptance traces for later work, not runtime tests.
- **Resolved authority correction:** SIM-I04 now projects both QAD-038 sources. Invite mode preserves the standard URL-selected flow and limits `Manage Guild` to permission bits the installer holds. Code-defined mode binds consent/commit to the application's authenticated saved revision/hash and lets `Manage Guild` approve exactly that set even when one bit is unheld. Either committed set becomes the durable ceiling; caller-selected additions, position/supplemental roles, ceiling reset, and later manager-only expansion are rejected. Preview/audit remain visibility controls, while source/equality/revision checks and the retained ceiling enforce the boundary.
- **Corrections caught by simulation and now resolved:** QAD source citations use `guild/services/member/GuildMemberRoleService.ts`; the reconciler uses QAD-149's `INSTALL_CLEANUP_FAILED`; rollout uses the typed gate names; message research now distinguishes the internal nonconditional version field from the planned public concurrency-safe CAS; QAD assigns manager-Forget/expiry markers to the non-resurrection journal; reinstatement excludes verified-dormant uninstalls; component reservations differ from consumption; and dedicated projections now cover builders/native actions, legacy and permanent deletion, policy, creation/handle limits, personalization, excluded contexts, native parity/collisions, all authority modes, rate saturation, autocomplete, modals, pre-parse amplification, and capability negotiation. HTTP endpoint verification is distinct from the explicit transport-mode switch, whose stale-write, queued-work, clearing, and signing-key branches are projected separately. The final reference scan confirms the superseded operative forms are absent.
- **Fresh contract/coverage correction:** capability syntax now accepts the permanent hyphenated IDs; active and pending HTTP endpoint configurations have separate identities/pointers; command error codes are canonical; causal application audit rows cannot be destructively compacted; and current version/component evidence no longer implies planned CAS/ownership already exists. SIM-P03, A04, E02, C10, C11, and S04 add the previously missing audience, confirmation-class, report-purge, handle-freeze, staged-draft, and generic message-template chains.
- **Closure contract/trace correction:** transport and verification-key settings are current-owner-only; endpoint candidates persist proof through a candidate-local conditional update while only the application-pointer CAS activates it; native operations use their canonical dotted IDs; component traces commit only on the selected owner's terminal success; generic message creation includes ownership; and a stale activation CAS does not itself abandon a candidate or delete key material. The later bounded closure review superseded an earlier no-gap claim by adding the guild-route single-schema portability chain in SIM-C12 and the stale-`INSTALLING` crash-abandonment branch in SIM-I02.
- **Common-journey coverage:** SIM-G01 connects creation, invite-mode install, compatible registration, selection, Gateway delivery, public response, retry, and revocation. SIM-G02 connects the separate code-defined install and structured-action opt-ins through exact consent, confirmation, execution-time reauthorization, causal audit, and later ceiling enforcement. The common-example index routes other reader questions to the existing detailed failure/concurrency traces.
- **Deployment-owned unknowns:** RPO, RTO, backup retention, restore-drill cadence, geographic replication, alert thresholds, traffic-dependent observation windows, and safety margin have no repository-derived values (QAD-192/QAD-221). SIM-R01 and SIM-O01 deliberately stop a launch/recovery gate when these records are absent.
- **External repository boundary:** no Flutter client is present. Unsupported-client simulation validates static/inert fallback in current shared/React scope but cannot claim native Flutter execution validation.
- **Evidence-gated UI:** QAD-236 intentionally leaves the final collision grouping, any extra provider-cycling shortcut, and preferred-provider control visual unselected. SIM validates fixed identity/ordering/accessibility invariants and blocks production selection until the required real-composer evidence and QAD addendum exist.
- **Deferred project boundary:** QAD-237 permits no SDK work in the current eleven trains. SIM/conformance outputs may later become evidence for a separately authorized thin TypeScript project; they do not create or publish that package.
- **Legacy report policy:** QAD-213 gives pre-activation resolved database/search rows a `LEGACY_RETAINED` marker and infers no new destructive deadline pending a separately reviewed migration. That marker does not extend object snapshots beyond the current public up-to-one-year lifecycle or any binding legal obligation.
- **No silent policy choice:** QAD-236 collision-control selection, exact selected operation-adapter rollout thresholds, and deployment numerical gates remain inputs, not values supplied by SIM. QAD-038 is no longer an input: SIM applies its exact-first-install and owner/`Administrator`-expansion answer.
