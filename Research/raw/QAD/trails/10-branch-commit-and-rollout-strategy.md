# Branch, Commit, and Rollout Strategy Trail

## Record status

Structured from the recovered 2026-07-17–20 `Analyze bot commands and roles` task and completed with user-authorized repository-derived implementation details on 2026-07-18. The exact strategy is in `specs/stacked-branch-pr-and-rollout-strategy.md`; `../provenance.md` classifies decision authority. Launch-blocking product/operations choices tracked elsewhere still have to be answered before the affected train starts.

## Current Fluxer basis

- Fluxer is a pnpm monorepo with TypeScript packages/services plus Rust and Erlang services. Bot-platform contracts cross `packages/schema`, `fluxer_api`, `fluxer_gateway`, `fluxer_messages`, and `fluxer_app`.
- Database, shared-schema, OpenAPI, Gateway, and client changes therefore cannot safely be treated as isolated UI pull requests.
- Repository instructions require inspecting generated sources and using the documented generator rather than manually editing generated artifacts.
- No implementation branch or commit currently exists for this work.

## Accepted strategy so far

- Use short dependency stacks, normally no more than two or three open dependent implementation pull requests.
- Avoid one permanent mega-stack spanning every future feature.
- Keep delegated user authority in a separate high-risk stack.
- Store immutable command-manifest revisions in the base registry, but ship developer-facing staged publication and recovery later (`QAD-115`).
- Keep the separate Flutter client outside these branches; responsive React work remains in scope.
- Complete all planning questions and share the entire master plan in Fluxer's developer server before beginning any implementation branch; material maintainer feedback returns to the plan first.

## Completed repository-derived strategy — QAD-202

- **Question:** What exact dependency DAG, branch/PR scopes, commit boundaries, dark rollout, rollback, generator/test gates, rebase procedure, and documentation/SDK placement fit Fluxer's current repository?
- **Current precedent:** Fluxer is a cross-language pnpm monorepo with root aggregate checks, Cassandra/OpenAPI/schema generators, linear squash-style PR history, Conventional Commit PR-title validation, public API docs, generated downstream SDK signaling, no RFC directory, and no Flutter tree.
- **Recommendation and answer:** use eleven short merge trains, never more than three dependent PRs open, beginning with real database CAS and ending with separately gated delegated authority. Land tolerant readers before producers, use QAD-191 private flags, keep safety/reconcilers through rollback, commit generated artifacts with sources, run root plus boundary-specific checks, rebase descendants with range-diff/force-with-lease discipline, and ship docs/OpenAPI with each public surface rather than a final dump. Exact branch names, commits, acceptance, gates, and merge procedure are in `specs/stacked-branch-pr-and-rollout-strategy.md`.
- **Why this is sound:** the accepted platform crosses persistence, generated contracts, API, Gateway, workers, and UI, so ordering tolerant readers and dark prerequisites before writers is necessary for mixed-version safety. Short trains cap review/rebase cost, while preserving revocation, suspension, cleanup, and reconciliation during producer rollback prevents deployment controls from weakening security or data lifecycle.
- **Classification:** preserves current repository/tooling/review conventions and extends them with the minimum cross-service rollout discipline; it replaces the earlier coarse six-stack sketch.

## Deferred official TypeScript reference SDK — QAD-237

- **Question:** Does the newly approved SDK direction enter QAD-202's implementation trains now, and what authority, stability, scope, support, compatibility, security, and release gates govern any later official library?
- **Options considered:** add an SDK to the current dependency DAG; make an SDK mandatory for bot migration; generate broad multi-language clients immediately; let a TypeScript library define behavior; or preserve the raw-protocol trains and authorize one later thin reference project after contract stability.
- **User-approved answer:** QAD-202's eleven trains remain unchanged and contain no SDK package/publication. Compatible Discord libraries and raw HTTP/Gateway use remain supported and do not depend on a Fluxer SDK. After public protocol/schema/OpenAPI/compatibility-manifest stability, clean shared cross-runtime fixtures, and passing real-service raw plus pinned-Discord smoke evidence, maintainers may separately review and authorize a thin official TypeScript reference project. No other language deliverable is promised.
- **Authority and scope:** public schemas, compatibility manifest, service behavior, and shared fixtures are authoritative; SDK code/snapshots are consumers. Initial scope is generated public types, command/schema builders and local validation/canonicalization, explicit hash-aware registration, signature verification/parsing, deadlines and response/defer/follow-up/autocomplete/modal helpers, capability/fallback handling, and fixture-backed test utilities. Raw protocol use remains documented.
- **Safety boundaries:** import/construction performs no network request; auto-registration requires explicit observable application action; helpers expose validation/conflict/rate/deadline/idempotency truth and do not hide unsafe retries. The library never reproduces or claims authority over server availability, community policy, audit persistence, native permission checks, delegated capability issuance, durable idempotency, lifecycle, or final action authorization. Default logs/diagnostics exclude secrets, signatures/tokens, arguments/values, private responses, and arbitrary bodies.
- **Official-support gate:** assign maintainers/security response, supported Node/runtime and protocol ranges, semantic version/deprecation policy aligned with QAD-215 overlap, compatibility matrix, release signing/provenance, dependency-update process, and disclosure channel before using `official`/`supported`. The package must consume the same raw fixtures, pass raw-versus-SDK parity and real-service smoke tests, and preserve generated-source ownership. Publication is a separately authorized external action.
- **Current and supplied evidence:** QAD-196/QAD-220 already make raw fixtures and pinned `discord.js` the compatibility authorities; QAD-202 explicitly excludes an invented official SDK from current trains. `Research/CS/repository.md` verifies private schema/protobuf/OpenAPI generators but no official command SDK package or TypeScript client generator. The PDF recommends an SDK but supplies no maintenance/version/security or protocol-authority plan.
- **Why this is sound:** deferral prevents an unstable convenience layer from becoming an accidental specification or multiplying change work. A thin generated/fixture-backed consumer can later improve ergonomics while raw parity protects other languages and migrated Discord bots. Explicit maintenance/security gates avoid calling an unowned package official.
- **Compatibility classification:** `compatible_additive` only if/when separately shipped. The present decision changes no public endpoint or migration requirement. A future SDK must preserve every compatibility-manifest classification rather than redefining it.
- **Risks and unknowns:** protocol stability is an evidence judgment, not a calendar date; generated/manual code can drift; dependency/security maintenance is ongoing; and users may mistake helpers for server authorization. Separate review, parity CI, prominent boundary documentation, and assigned ownership are mandatory.
- **Downstream contracts:** add `specs/official-command-sdk.md`; clarify registration, interaction, error/rate, observability, conformance, documentation, and rollout contracts. The SDK remains outside SIM for current trains except that SIM/conformance outputs become its later input gates.
- **Authority and currency:** the user approved QAD-237 on 2026-07-20. The PDF is supporting product input only and does not authorize package creation or publication.

## Human review gate — QAD-214

- **Question:** May implementation branches begin once this research corpus is internally complete, or must project maintainers review the whole cross-service plan first?
- **Options considered:** begin product work immediately; seek review piecemeal in implementation pull requests; or share the complete decision/evidence/specification/DAG package before any bot-platform implementation branch and reconcile material feedback first.
- **Accepted answer:** require complete-plan review in Fluxer's developer server before implementation. Conflicting product intent returns to decision-making; maintainers, not automation, own public discussion and pull-request prose with truthful LLM disclosure.
- **Why sound / evidence:** the plan adds public contracts, persistence, privileged authority, privacy retention, cross-runtime schemas, migrations, and eleven merge trains. Repository `AGENTS.md` forbids automated pull requests/comments and `.github/CONTRIBUTING.md` documents the human contribution process. Deferring the first holistic review to dependent PRs would make architectural correction more expensive and fragment security/product context.
- **Tradeoffs, assumptions, and unknowns:** review delays coding and may reopen accepted planning records, but reduces rework and ensures project ownership. Reviewer availability, venue logistics, and feedback timing are external; this decision authorizes neither automated posting nor implementation.
- **Dependencies / consequences / supersession:** depends on a complete QAD/CS/SIM package and QAD-202's DAG. No product-code branch begins until the review occurs and material feedback is reconciled. The review must verify the resolved QAD-038 boundary before Train I2 implements authority writes. It supersedes any inference that planning completion alone authorizes implementation; it does not supersede later per-PR review.
- **Classification:** accepted process decision and explicit external human gate, not a product behavior or implementation recommendation.

## Cross-cutting completeness audit

- **Scope:** supplements QAD-202, QAD-214, and QAD-237; QAD-115/QAD-191/QAD-196/QAD-215/QAD-216 through QAD-220/QAD-221/QAD-222 are dependencies or rollout inputs, not branch-strategy decisions owned here.
- **Shared credible alternatives and rejection:** use one long-lived mega-stack; open every dependent change simultaneously; ship isolated UI/service pull requests before shared contracts; deploy writers before tolerant readers; roll database schemas backward; defer docs/generators/tests to a final cleanup; keep rollout flags indefinitely; or begin implementation before holistic human review. These alternatives increase review drift, incompatible mixed-version behavior, irreversible rollback, generated-source divergence, operational debt, and late architectural rework. Short prerequisite-ordered trains, dark additive contracts, producer rollback with readers/safety retained, per-surface documentation, and a pre-implementation review gate address those risks.
- **Evidence-backed soundness:** the repository is a pnpm TypeScript/Rust/Erlang monorepo with root checks, schema/OpenAPI/Cassandra generators, cross-service contracts, squash-style history, Conventional Commit validation, public API docs, and no Flutter/RFC tree. Those facts support short cross-service dependency trains and generator-aligned commits. They do not prove every planned product decision; QAD-214 requires maintainers to review the complete artifact first.
- **Tradeoffs:** security isolates delegated authority and keeps suspension/reconcilers during rollback; operations gain dark rollout/recovery gates but carry temporary dual readers/flags; compatibility lands tolerant consumers before producers and docs with contracts; maintenance limits open dependency depth but requires rebases/range-diffs and disciplined cleanup; users see slower staged delivery but lower mixed-version and rollback risk.
- **Assumptions and unknowns:** branch/PR creation, force-push, review, and public communication remain separately authorized human actions; exact release windows and deployment gates are external; maintainer feedback may reopen this DAG; generated command names must be rechecked against repository tooling at implementation time. QAD-038 is a fixed product/security boundary for I2, not branch mechanics. QAD-226's research branch is not one of these product trains.
- **Consequences and dependencies:** QAD-202 orders eleven trains, limits dependent PR depth, defines dark/rollback/generator/test/docs boundaries, and depends on the accepted domain contracts. QAD-214 blocks all product implementation until QAD/CS/SIM and deployment-owned gaps are presented and material feedback reconciled. QAD-237 leaves that DAG unchanged and permits only a later separately reviewed SDK project after protocol/conformance/support gates.
- **Supersession:** QAD-202 replaces the earlier coarse six-stack sketch. QAD-214 supersedes any implied authorization to start merely because the plan is complete. QAD-237 clarifies QAD-202's “no official SDK” as “none in these trains,” without adding a train or authorizing publication. None authorizes Git/public actions beyond QAD-226's separate research publication.
