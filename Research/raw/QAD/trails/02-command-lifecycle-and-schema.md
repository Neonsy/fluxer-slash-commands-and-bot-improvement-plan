# Command Lifecycle and Schema Trail

## Record status

Structured from the recovered 2026-07-17–20 `Analyze bot commands and roles` task, the accepted decision index, repository inspection, and the earlier Discord documentation comparison. QAD-227 through QAD-229 and QAD-232/QAD-233 were appended on 2026-07-20 from explicit user approval. This is not a verbatim transcript; `../provenance.md` classifies decision authority.

## Current Fluxer basis

- `fluxer_app/src/features/devtools/hooks/useCommands.ts`, `fluxer_app/src/features/devtools/utils/CommandUtils.ts`, and `fluxer_app/src/features/messaging/utils/SlashCommandUtils.ts` define and parse Fluxer's current native commands in the client.
- Current command recognition includes hard-coded names such as `/ban` and `/kick`; there is no application-owned server command registry, immutable command key, published manifest, or per-community command configuration.
- Application records and OAuth bot authorization do not currently store command definitions or revisions.
- Consequently, lifecycle, persistence, schema publication, and application command versioning are new server-owned contracts. Existing native commands supply UX and authorization precedent, not reusable persistence.

## Decision trail

### Stable command identity and dormant configuration — QAD-021, QAD-027, QAD-028, and QAD-029

- **Question:** Does community configuration follow a mutable name, declaration order, a platform-generated ID, or a stable developer key?
- **Risk:** name or order reuse can attach old permissions and aliases to a different command.
- **Recommendation and answer:** every command has an immutable developer-defined key scoped to its application. Community settings follow that identity, have no automatic time-based expiry while their application/community owners exist, and reactivate only if the same key returns. Authorized reset or permanent application/community deletion removes them; a same-name new key inherits nothing.
- **Why this is sound:** no current application-command store exists from which to derive a safe expiry interval, while current user and per-community settings are lifecycle-owned persistent rows rather than inactivity-TTL data. An invented timer would silently erase community-owned policy even though the same stable command may legitimately return after a long absence. Explicit reset and application/community deletion provide deterministic owners; measured storage or privacy evidence may justify a separately reviewed bound later.
- **Classification:** new persistence contract; no current application command identity exists.

### Developer and community disablement — QAD-022, QAD-023, and QAD-030

- **Question:** Is disabling destructive, and can a community override developer disablement?
- **Options considered:** delete settings; keep dormant settings; let community force-enable globally unavailable commands.
- **Recommendation and answer:** developer and community disablement are non-destructive. Developer disablement always wins; community enablement becomes effective only after the developer re-enables the same command key.
- **Why this is sound:** the developer owns whether code is offered, while the community owns whether it is allowed locally; neither disable action needs to destroy the other's configuration. This keeps rollback reversible without letting a community invoke code the developer has withdrawn.
- **Classification:** new layered availability model.

### Reset behavior — QAD-024, QAD-025, and QAD-026

- **Question:** Who may reset command configuration, at what scope, and what does reset remove?
- **Recommendation and answer:** owner, `Administrator`, or `Manage Guild` may reset one command or every command for an application. Reset removes every community-owned override but cannot defeat developer policy. Application-wide reset requires explicit confirmation.
- **Why this is sound:** these roles already own community application configuration, and reset affects only that owner's state. Scoping and confirmation make broad deletion deliberate, while preserving developer policy prevents reset from becoming an authorization bypass.
- **Classification:** new community administration behavior aligned with the stronger application-management authority boundary.

### Permanent application deletion — QAD-031

- **Question:** Should dormant configuration outlive the application identity itself?
- **Recommendation and answer:** no. Permanent global application deletion removes active and dormant configuration because its owner identity cannot legitimately return.
- **Why this is sound:** dormant data is useful only while the same immutable application can be reinstalled. Once that identity is permanently deleted, retention has no valid future owner and would create orphaned configuration or risk attachment to a different application.
- **Classification:** extends current application deletion to new configuration data.

### One schema per command key — QAD-109

- **Question:** May one stable command key publish different definitions in different communities?
- **Tradeoff:** per-community schemas enable customization but destroy predictable identity, documentation, versioning, and recovery.
- **Recommendation and answer:** one application-wide schema per key. Community policy layers availability, aliasing, and restrictions without forking the definition.
- **Why this is sound:** one immutable key must describe the same inputs and behavior wherever it is invoked for documentation, stale-form validation, and manifest recovery to remain reliable. Community needs are still served through policy that does not redefine the command.
- **Classification:** new registry invariant.

### Community targeting — QAD-110

- **Question:** How may developers stage availability without per-community schema forks?
- **Recommendation and answer:** a command may target all installed communities, selected installed communities, or none. Targeting changes availability only.
- **Why this is sound:** targeting provides staged rollout and testing while preserving the one-schema invariant. Keeping it separate from definition and community approval prevents a target edit from silently changing inputs or local authority.
- **Classification:** new developer rollout control.

### Auto-enable and authority expansion — QAD-111

- **Question:** Should a new or newly targeted command become usable automatically?
- **Options considered:** always enable; always require review; enable only within authority already approved by that community.
- **Recommendation and answer:** auto-enable only when execution mode, native operations, permissions, and other authority remain inside the existing approval envelope. Any expansion remains pending the applicable review and is surfaced to relevant managers. Bot permission, hierarchy-position, or supplemental-role expansion requires owner/`Administrator`. Under QAD-041, owner/`Administrator` also reviews a mode transition that introduces bot authority or removes the invoking-user check from bot-authority execution; `Manage Guild` may approve only the other enumerated mode transitions and non-bot-authority application configuration explicitly assigned to it elsewhere.
- **Why this is sound:** automatic availability is low-friction when it cannot increase granted capability, but an expanded authority envelope is a new community security decision. Comparing exact structured authority avoids both needless reapproval and silent escalation.
- **Classification:** new least-surprise approval behavior built on current explicit bot-permission grants.

### Automatic structural versions — QAD-112

- **Question:** Must developers manually manage a schema version, and what happens to stale clients?
- **Recommendation and answer:** Fluxer automatically derives and increments structural schema versions. Stale submissions are rejected before application delivery with a structured refresh path; compatible entered values are retained where possible. Metadata-only edits do not invalidate an open command, and already-created interactions keep their submitted version.
- **Why this is sound:** the platform can deterministically detect shape changes and therefore cannot rely on developers to increment a safety-critical counter correctly. Reject-and-refresh prevents old fields from being reinterpreted, while keyed same-type retention minimizes unnecessary user re-entry.
- **Classification:** new platform-owned consistency mechanism.

### Immediate and staged publishing — QAD-113

- **Question:** Require only Discord-style immediate registration or only atomic manifests?
- **Recommendation and answer:** preserve compatible immediate publishing for migration and add an optional complete-draft validation and atomic-publish workflow.
- **Why this is sound:** immediate endpoints let existing developer workflows migrate, while a validated complete candidate prevents large releases from exposing partial intermediate state. Both paths can publish the same immutable manifest, avoiding two registry models.
- **Classification:** additive extension above the Discord-compatible baseline; no current registry exists.

### Manifest recovery — QAD-114 and QAD-115

- **Question:** Should rollback restore prior definitions, and must it block the initial registry?
- **Recommendation and answer:** store immutable published revisions from the first registry. Later recovery republishes an old revision's contents as a new immutable revision and revalidates approvals. It cannot roll back backend code, application data, external effects, or completed interactions. Publishing and recovery ship as later focused stacked pull requests, not on the base interaction dependency path.
- **Why this is sound:** immutable history makes a definition recovery auditable and prevents rewriting what past interactions used; republishing through current validation avoids reviving obsolete authority. Deferring the recovery UI/API keeps a nonessential control from blocking the core registry.
- **Classification:** new recovery facility with deliberately bounded semantics.

### Initial option schema — QAD-116

- **Question:** Start with a minimal bespoke schema or the Discord chat-input option baseline?
- **Recommendation and answer:** support subcommands, subcommand groups, string, integer, boolean, user, channel, role, mentionable, number, and attachment values, plus compatible choices, bounds, and autocomplete. Encode the schema as a versioned tagged union so Fluxer extensions remain possible with capability/fallback rules.
- **Why this is sound:** the documented Discord set covers established migration needs and avoids an immediately incompatible bespoke subset. A bounded tagged union gives exhaustive validation and safe additive evolution instead of accepting ambiguous arbitrary JSON.
- **Classification:** new implementation designed for migration compatibility without making compatibility a ceiling.

### Command-type boundary — QAD-117

- **Question:** Include user-context, message-context, or Activity entry commands?
- **Recommendation and answer:** no; only chat-input application commands belong to the plan.
- **Why this is sound:** chat input has a defined option, submission, and interaction model in this plan; the excluded entry types require different selection context, privacy, and authorization contracts. Excluding them prevents unreviewed semantics while leaving future tagged-union extension possible.
- **Classification:** explicit non-goal; current native/context behavior is not removed.

### Query-first command persistence — QAD-165

- **Question:** What source rows, query tables, heads, and indexes store stable commands, immutable manifests, optional drafts, targeting, community configuration, and user preferences?
- **Options considered:** relational-style secondary indexes; mutable command rows without publication snapshots; query-first immutable revisions selected through a CAS head.
- **Recommendation and answer:** use stable command/key mappings, immutable command revisions and manifest entry partitions, one versioned manifest head, one optional versioned draft, two targeting query directions, community/application configuration partitions, and preference tables matching their accepted account/community scopes. Publication exposes a complete candidate only by advancing its head. Full fields and access paths are in `specs/command-persistence.md`.
- **Current precedent:** `fluxer_api/src/api/Tables.ts` defines explicit query tables such as `applications` plus `applications_by_owner`; `fluxer_api/src/api/oauth/repositories/ApplicationRepository.ts` owns both and uses versioned updates and batches. Fluxer does not rely on ad hoc secondary indexes, and current user/user-guild settings already own synchronized preferences at their respective scopes.
- **Why this is sound:** the tables match known read directions and put source/index repair under one repository owner, following Fluxer's storage conventions. Immutable candidates plus one conditional head make publication atomic to readers and support recovery without mutating history.
- **Classification:** extends current query-first and versioned-repository conventions to a new command registry while adding immutable publication semantics required by staged validation and recovery.

### Command grammar and limits — QAD-166

- **Question:** Which command/key/name/description/option/localization/manifest limits ship initially?
- **Recommendation and answer:** retain the Discord chat-input limits (100 commands, 32-character names, 100-character descriptions, 25 options/choices, permitted two organizational levels, 6000-character strings, and 8000 counted command text), add a 64-character immutable developer key, and cap canonical definitions/manifests at 512 KiB/8 MiB. Target mutations page at 1000 guilds. Full rules are in `specs/command-schema-limits-and-localization.md`.
- **Current precedent:** Fluxer validation uses centralized code constants and Zod schema caps; it has no application-command limits. Discord's official application-command contract supplies the migration-compatible baseline.
- **Why this is sound:** preserving familiar semantic limits supports compatible clients, while explicit byte caps bound locale and canonical-serialization amplification that character counts alone miss. Central constants and paged targeting make enforcement testable and operationally bounded.
- **Classification:** adds a compatible schema plus Fluxer-specific serialized-size protection required for complete locale maps.

### Application-owned localization — QAD-167

- **Question:** What locale registry, validation, fallback, collision, search, and version behavior applies to application command text?
- **Recommendation and answer:** accept only Fluxer's current `AllLocales`, validate every value, require default text, use exact-locale then default fallback, reject per-locale effective name collisions, index locale-tagged plus default text, and treat localization-only publication as non-structural metadata history.
- **Current precedent:** `packages/constants/src/Locales.ts` and `packages/schema/src/primitives/LocaleSchema.ts` centrally define account UI locales, while Lingui catalogs own Fluxer UI text rather than third-party content.
- **Why this is sound:** one existing locale registry prevents unsupported tags and inconsistent fallback, while keeping application strings outside Lingui preserves content ownership. Effective-name collision checks ensure localization cannot create ambiguous invocation in any supported locale.
- **Classification:** extends the existing locale registry without conflating platform translations with application-owned strings.

### Native command adaptation — QAD-168

- **Question:** How can built-in commands share discovery with applications without moving trusted native behavior into third-party persistence or weakening API authorization?
- **Recommendation and answer:** adapt current code-owned definitions/handlers into a typed native `CommandDiscoveryEntry` provider and merge it with a distinct application provider in the picker. Preserve local transforms and current command modules; keep REST services authoritative; do not allow application settings or user third-party hiding to alter native entries. Never dispatch on a shared name alone.
- **Current precedent:** `fluxer_app/src/features/devtools/hooks/useCommands.ts`, `fluxer_app/src/features/messaging/utils/SlashCommandUtils.ts`, and `fluxer_app/src/features/devtools/utils/CommandUtils.ts` own current client discovery/parsing/dispatch, while guild member/moderation services independently enforce native permissions and hierarchy.
- **Why this is sound:** a shared typed discovery shape gives users one picker without moving trusted native handlers or authorization into third-party storage. Provider identity and structured selection prevent a same-name application command from intercepting native behavior.
- **Classification:** refactors discovery ownership while preserving observable native behavior and server authorization.

### Discord-compatible registration surface — QAD-169

- **Question:** Which standard endpoints ship, and how can immediate/guild-scoped writes coexist with immutable manifests and one application-wide schema?
- **Options considered:** Fluxer-only manifest endpoints; reproduce per-guild schema forks; expose compatible endpoints as immediate publication and community-target views over one schema.
- **Recommendation and answer:** support standard global/guild command CRUD and bulk-overwrite shapes for chat-input commands. Global writes publish schema; guild writes alter targeting and reject definition conflicts. Add optional immutable `key` and derive one from initial name for compatibility. Every successful write commits a complete immutable manifest through a head CAS.
- **Current precedent:** current OAuth applications have one owner and bot token, and repository helpers support versioned updates; no command endpoint exists. Discord's official application-command documentation defines the migration endpoint/status/upsert baseline.
- **Why this is sound:** mapping familiar routes onto one canonical manifest preserves developer migration while rejecting per-guild schema divergence that would break stable identity. Complete publication through a conditional head ensures even compatibility writes cannot expose partial state.
- **Classification:** adds a compatible surface while intentionally replacing per-guild schema variation with approved targeting semantics.

### Structural compatibility and stale-form mapping — QAD-170

- **Question:** Which changes invalidate input, and may stale values be coerced or submitted automatically after refresh?
- **Options considered:** treat every revision as breaking; guess identity from mutable names and coerce; add stable field keys and retain only validated same-type values.
- **Recommendation and answer:** distinguish command, schema, and authority revisions. Metadata changes do not invalidate. All validation/shape changes increment schema version. On stale submit, return current schema/diff, map exact keyed paths with unchanged types, revalidate, visibly clear invalid fields, and require resubmission. Never coerce or auto-invoke.
- **Current precedent:** Fluxer centralizes validation and carries integer version metadata, but its generic read/increment/update helper is not stale-write protection or monotonic CAS. Current client slash parsing has only mutable names and cannot safely preserve structured values; F1 supplies the conditional primitive for the planned registry.
- **Why this is sound:** separate revisions invalidate only the assumptions that changed, reducing needless disruption. Stable keys and same-type revalidation preserve safe input, while visible clearing and explicit resubmission prevent stale values from being coerced into a different command meaning.
- **Classification:** adds stable option identity and a conservative refresh contract required by automatic schema versioning.

### Command errors and registry limits — QAD-171 and QAD-172

- **Question:** Which typed failures and route budgets cover registry, publication, targeting, approval, context, and autocomplete behavior?
- **Recommendation and answer:** extend `FluxerError` with domain codes and audience-safe typed data; retain path validation and existing rate-limit headers. Reuse 60/min developer reads, 30/min developer mutations, 20/10-second guild mutations, the accepted autocomplete/delivery buckets, a 10/min expensive publication/target budget, and a compatible 200/day new-identity ceiling. Full matrices are in `specs/command-errors-and-rate-limits.md`.
- **Current precedent:** current errors use stable uppercase codes plus typed data, while OAuth developer and guild route configs already establish the reused budgets.
- **Why this is sound:** extending the existing envelope gives clients stable recovery instructions without leaking hidden targeting or suspension state. Reusing proven route-budget scales preserves operational expectations, while stricter publication/identity limits bound expensive validation and churn.
- **Classification:** extends current error/rate machinery rather than creating command-specific transport semantics.

### Bounded history — QAD-224

- **Question:** How are application-lifetime dormant command identities and immutable manifest history bounded without deleting another party's live configuration or making rollback unbounded?
- **Options considered:** retain every identity and manifest forever; time-expire inactive identities even while referenced; use only a byte cap; or combine count bounds with explicit, reference-safe retirement and a small pinned recovery set.
- **Accepted answer:** one application may retain at most 1,000 command identities, including inactive identities, and the newest 20 manifests plus at most 5 explicitly pinned older manifests. Temporary references may delay cleanup only while a live interaction/publication/recovery/reconciliation operation needs the exact data. An owner may retire an inactive identity only when no retained manifest, guild configuration, user preference, availability record, or live operation references it. Later reuse of the developer key creates a fresh command ID and inherits nothing.
- **Why this is sound:** the 100-active-command and 200-new-identities-per-day compatibility limits bound live and burst behavior but not lifetime tombstones or immutable history. Unbounded retention permits storage amplification; blind TTL pruning breaks QAD-021/QAD-028 community ownership and recovery. Explicit retirement makes deletion authority and safety conditions visible while preserving permanent Snowflake non-reuse.
- **Repository/external evidence:** Fluxer has no application-command registry from which a lifetime value can be derived. Its repositories use explicit query tables and owners rather than implicit garbage collection. Discord's current application-command contract documents 100 global chat-input commands and a 200-create-per-day-per-guild limit, but no Fluxer lifetime-history policy; `specs/command-persistence.md` therefore records these accepted product bounds and all reference checks.
- **Calibration and review:** 1,000 identities and 20+5 manifests are accepted product bounds, not values proved by current code or Discord. They must be enforced together with the existing 8 MiB canonical-manifest limit and measured per application for retained bytes, blocked retirements, pin use, cleanup delay, and legitimate cap rejections. Repeated normal-workload rejection or violation of the deployment's measured storage/reconciliation envelope triggers an explicit product review; implementation never silently prunes references or raises the counts.
- **Tradeoffs and unknowns:** storage/maintenance become bounded and owners can reclaim capacity, but long-lived referenced configuration can prevent retirement and pinned recovery is intentionally limited. Exact byte-level operational alarms and cleanup cadence remain implementation/deployment values within these hard counts.
- **Dependencies and consequences:** depends on QAD-021/QAD-027/QAD-028, QAD-114, QAD-165, and QAD-172. Persistence needs reference queries, a retirement preflight/result, cleanup reconciliation, and a new immutable command ID on key reuse; clients must never present retirement as ordinary disable/delete.
- **Supersession:** closes the durable-growth question formerly routed from `commands.md` and `question-routing.md`; it does not supersede the 100-active-command or 200/day limits.
- **Classification:** accepted product retention and deletion policy, not a current-code-derived conclusion.

### Multi-value resource options and `TARGET` — QAD-227

- **Question:** Should a command option accept several resources, and if so what identity, cardinality, wire, compatibility, authorization, stale-state, and privacy rules control it?
- **Options considered:** retain only the Discord scalar baseline; permit any option to become repeatable and overload compatible `value` with either a scalar or array; or add a bounded typed Fluxer extension for the resource-selection cases with explicit capability and fallback behavior.
- **User-approved answer:** add repeatable `USER`, `ROLE`, `CHANNEL`, and `MENTIONABLE` options and a repeatable Fluxer `TARGET` union over users, roles, and channels. Registration uses `multi_value: {min_values, max_values}`; `min_values` defaults to 1, `max_values` is required, and neither may produce more than 50 selections. `required` independently controls omission. Collections preserve user selection order and reject duplicates by typed resource identity.
- **Wire and discovery contract:** compatible scalar options keep `value`; repeatable options use ordered `values`. Single-kind collections contain IDs, while `MENTIONABLE` and `TARGET` contain `{type, id}` entries. Native resource search is the default. An application callback is opt-in and receives only the current option path and permitted ordered selections; it remains limited to 25 suggestions/64 KiB. The complete interaction remains under 512 KiB. Commands requiring the extension are omitted from ordinary discovery on an unsupported client, while exact lookup/deep link returns an update-required state; no client may coerce a collection to one value.
- **Validation and authority:** submission revalidates schema, min/max, duplicates, current command access, and every resource's visibility/selectability. A stale, deleted, inaccessible, or otherwise invalid member rejects the entire submission before application delivery; the client preserves valid selections, identifies safe corrections without confirming hidden-resource existence, and requires resubmission. Selection is never authorization. Applications still enforce their own action semantics, and any Fluxer-native operation performs its normal current-state permission, hierarchy, target, and capability checks.
- **Evolution, accessibility, and privacy:** scalar/multi shape changes, allowed target-kind changes, and bound changes increment `schema_version`. Stale collections are never coerced or silently truncated and are retained only when the same stable option key, type, and current bounds accept them. Search, selection, review, removal, counts, focus, and errors are release-gated on complete keyboard and screen-reader operation. Selected IDs/values never enter logs, traces, analytics, or command history; only non-identifying cardinality and error classes may support capacity monitoring.
- **Current and supplied evidence:** `Research/CS/commands-and-permissions.md` records a fixed client-owned native command list and hard-coded parsing rather than an application registry or repeatable public option schema. QAD-116 already reserves a versioned tagged-union/capability path, QAD-166 supplies serialized bounds, QAD-170 supplies conservative stale mapping, QAD-189 supplies negotiated extension identity, and QAD-209 separately treats multiple-target native effects as confirmation-sensitive. The supplied five-page `fluxer_slash_commands_specsheet.pdf` advocates multi-selection, the four compatible resource types, `TARGET`, chips, and an approximate 50-value starting point, but labels the hard limit open.
- **Why this is sound:** the selected form solves the supplied resource-batch use cases without changing any Discord scalar field or surprising a migrated handler with an array. Typed identity, all-or-nothing delivery, server access checks, application/native reauthorization, and non-coercing stale behavior prevent display labels, partial validation, or an old client from retargeting an action. Fifty is small relative to the existing byte envelope while still requiring boundary/load evidence before rollout.
- **Compatibility classification:** `compatible_additive`. An ordinary Discord registration and interaction remains unchanged. The Fluxer fields and `TARGET` are intentionally not directly registrable with Discord; a cross-platform application uses the scalar baseline there and advertises the Fluxer capability where supported.
- **Risks and unknowns:** 50 is an approved product starting bound, not a capacity derived from current Fluxer traffic or code. Application autocomplete reveals permitted current selections before final submission when a developer explicitly opts in. The richer composer and two-sided capability path increase implementation and accessibility complexity, and a multi-target command can increase action impact even though invocation itself grants no authority. Load tests, assistive-technology tests, compatibility fixtures, and QAD-209 confirmation tests therefore gate rollout.
- **Downstream contracts:** update `specs/command-schema-limits-and-localization.md`, `command-registration-api.md`, `command-schema-evolution.md`, `interaction-envelope.md`, `interaction-capability-registry.md`, `command-errors-and-rate-limits.md`, `command-picker-and-settings-ui.md`, `bot-platform-abuse-and-resource-protection.md`, `application-audit-contract.md`, `discord-compatibility-conformance.md`, and the existing C/E/P train acceptance criteria. Later SIM work must cover invalid/stale collections, capability fallback, permission changes, limits, and no partial invocation.
- **Authority and currency:** the user approved each material subdecision and then QAD-227 as a whole on 2026-07-20. The PDF is useful product input, not normative authority: its metadata identifies only a community draft dated 2026-07-10, with no author, revision identifier, citations, or publication channel. QAD-227 is authoritative because of the recorded user approval and its reconciliation with existing QAD/CS, not because the PDF proves current Fluxer intent.

### Conditional command-option relationships — QAD-228

- **Question:** May command fields declare value-dependent requirement, visibility, or exclusion, and what bounded grammar, graph validation, client/server behavior, stale handling, compatibility, and accessibility rules control it?
- **Options considered:** keep all cross-field validation application-only; attach ad hoc rule fields to individual options; use expression strings or a general JSON-logic tree; group per-kind maps; or store a single typed relationship graph beside each sibling option list. Inline fields make symmetric conflicts one-sided or duplicated, per-kind maps fragment schema/error handling, and expression languages introduce coercion, precedence, parser, and versioning complexity.
- **User-approved answer:** add one Fluxer-only `relationships` array beside each command/subcommand's sibling `options`. Initial entries are `required_if`, `visible_if`, and symmetric `conflicts_with`. Required/visible entries name one affected stable option key and one `when` predicate; conflicts name the two keys as one unordered pair. Keys must be siblings and relationships never cross subcommand branches.
- **Predicate and cardinality contract:** predicates are limited to presence (`is_set`/`is_unset`) or exact typed equality with one definition-declared eligible scalar. They exclude regex, ranges/order, resource IDs, collection membership/search, scripts, and nested `AND`/`OR`. Each option has at most one `required_if` and one `visible_if`, avoiding implicit Boolean composition; it may conflict with any of its other at-most-24 siblings.
- **Registration and runtime contract:** registration rejects self/missing/cross-branch references, duplicate conditions or conflict pairs, dependency cycles, and statically reachable states that make a field both required and hidden or otherwise impossible to satisfy. The client evaluates accepted rules live only as guidance. The server reevaluates every applicable rule from the published schema and submitted values; any missing conditional value, hidden populated value, or conflict rejects the complete submission before application delivery.
- **UX and errors:** false `visible_if` hides an empty option, but a populated option remains visible in a correction state until the user removes it. `required_if` changes only requiredness; false leaves an otherwise visible option optional. `conflicts_with` disables an empty peer and marks both populated peers without clearing either. Rules have no runtime precedence: all must pass. A bounded, audience-safe error list follows option declaration order, focuses the first correction, never echoes hidden values, and never silently clears, coerces, or partially delivers input. Every state is keyboard-operable and announced to assistive technology.
- **Evolution and compatibility:** every relationship change increments `schema_version`; stale forms reapply the graph, preserve only valid keyed input, identify corrections, and require explicit resubmission. The accepted interaction payload remains the normal scalar/QAD-227 collection option shape, so Discord handlers are unchanged. `fluxer.commands.conditions.v1` gates the Fluxer composer behavior; unsupported clients omit the command ordinarily and render an inert update-required exact result rather than ignoring rules.
- **Current and supplied evidence:** `Research/CS/commands-and-permissions.md` records hard-coded client parsing/autocomplete without a public application option schema or declarative relation graph. QAD-116 supplies the extensible tagged schema, QAD-170 the stable-key stale-form contract, QAD-195 the bounded validation/error posture, and QAD-220 the unchanged compatible handler baseline. The supplied PDF proposes `required_if`, `show_if`/`hide_if`, and `mutually_exclusive_with`, with live client guidance and human errors; QAD-228 normalizes those ideas into fewer unambiguous kinds and retains server authority.
- **Why this is sound:** a single typed sibling graph makes symmetric edges, graph validation, canonical hashing, generated types, and path errors uniform. Stable keys survive display renames. The deliberately small predicate language covers the supplied examples without creating a second executable policy engine, while server reevaluation and no-silent-clear behavior prevent manipulated/old clients from changing accepted meaning.
- **Compatibility classification:** `compatible_additive`. Definitions without `relationships` and every delivered option value preserve the Discord-compatible contract. Relationship fields are Fluxer registration data and are not sent to Discord; capable cross-platform tooling maintains or generates the Fluxer variant explicitly.
- **Risks and unknowns:** conditional UI increases composer and assistive-technology state complexity; overly connected definitions can be confusing despite structural validity; equality constants are public schema rather than secrets; and older clients cannot safely invoke a conditional form. Boundary, cycle, satisfiability, stale-state, manipulated-client, visual, keyboard, and screen-reader tests gate rollout.
- **Downstream contracts:** update the existing schema/limits, registration, persistence, evolution, picker, interaction preflight, capability, error/resource, compatibility, and C/E/P rollout specifications. Later SIM-C03 must cover graph rejection, stale forms, overlapping errors, manipulated clients, and fallback without inventing new condition policy.
- **Authority and currency:** the user approved the recorded constraints, selected the centralized array over credible alternatives, approved no-precedence error handling, and approved QAD-228 as a whole on 2026-07-20. The five-page community draft remains non-normative and carries no author, revision identifier, citations, or publication channel; its dated suggestions are input, not decision authority.

### Typed temporal command options — QAD-229

- **Question:** Which date/time/duration meanings become native command types, and what exact serialization, timezone, daylight-saving, bounds, relative entry, server validation, compatibility, stale-state, accessibility, and privacy contracts control them?
- **Options considered:** retain only application-parsed strings/integers; use one generic datetime whose locale/offset changes its meaning; expose free-form natural-language parsing; or define separate canonical calendar, instant, zoned-local, and elapsed-duration types with bounded client helpers and server validation.
- **User-approved answer:** add Fluxer `DATE`, `INSTANT`, `LOCAL_DATETIME`, and `DURATION`. `DATE` is `YYYY-MM-DD` with no time/zone in its value. `INSTANT` is UTC RFC 3339 at millisecond precision. `LOCAL_DATETIME` starts with `YYYY-MM-DDTHH:mm:ss.SSS` and resolves to a delivered tuple containing local value, IANA zone, chosen UTC offset, and exact UTC instant. `DURATION` is a nonnegative JavaScript-safe integer millisecond count. Locale changes presentation only.
- **Bounds and server time:** date/time years are `0001–9999`; applications may declare tighter same-type min/max bounds; duration has no unsupported universal product cap below the safe-integer representation. Parsing, calendar validity, bounds, and overflow reject rather than clamp. An option may declare `ANY`, `PAST`, or `FUTURE`; server submission time is authoritative. Instants/resolved locals compare exact instants, while dates compare the current calendar date in their resolved zone; past/future are strict.
- **Timezone and daylight-saving contract:** a command-fixed IANA zone wins; otherwise explicit picker selection, saved account zone, then community zone apply. If none exists, the user must choose; locale/device settings never silently decide. A daylight-saving gap is rejected without shifting. An overlap requires earlier/later selection, and the selected offset/resolved instant are delivered. If current server timezone data resolves a stale preview differently, Fluxer requires visible reconfirmation instead of substituting the new instant.
- **Relative input:** constrained client helpers may accept unambiguous forms such as `in 2 hours` or `2h 30m`, but must show the exact normalized value/zone before acceptance. The server/application receives only canonical values. Free-form phrases such as `next Friday evening` are unsupported, so locale-specific NLP never becomes hidden wire meaning.
- **Compatibility and evolution:** types `1001–1004` and `fluxer.commands.temporal.v1` are additive Fluxer contracts requiring both a capable client and application delivery transport. Discord definitions/handlers remain unchanged when omitted; cross-platform bots register explicit scalar alternatives, never a server-coerced fallback. Type, precision, timezone requirement, bound, or time-direction changes increment `schema_version`; stale input is retained only when same-type/currently valid, and changed local resolution requires reconfirmation. Display/helper-copy-only edits remain metadata.
- **Accessibility and privacy:** native pickers and manual entry must expose canonical confirmation, zone, overlap choice, bounds, and errors through keyboard/screen-reader semantics on desktop and narrow layouts. Raw temporal values/zones never enter logs, traces, analytics, or command history; only aggregate type/error classes are observable. A later native operation may audit its own normalized expiry/duration under the existing allowlist, never by copying raw command input.
- **Current and supplied evidence:** `Research/CS/commands-and-permissions.md` records only hard-coded local command parsing, with no application registry or native temporal option type. It also verifies an IANA-zone account/profile precedent that is staff-only and a complete current guild row with no community timezone, so ordinary baseline users have no stored-zone fallback and must choose explicitly. QAD-116 permits typed additive options, QAD-166/170 provide bounds and stale mapping, QAD-189 provides capability identity, and existing audit specs already prohibit raw option values. The supplied PDF asks for native date/datetime/duration pickers but does not distinguish instant/local semantics, define wire forms, resolve daylight-saving behavior, or set limits.
- **Why this is sound:** separate types prevent locale, timezone, and unit ambiguity at the API boundary. Canonical values plus server resolution make cross-runtime fixtures possible; explicit gaps/overlaps and no-clamping/no-coercion prevent silent schedule changes; the capability/fallback boundary preserves ordinary Discord migration without pretending its scalar types carry richer meaning.
- **Compatibility classification:** `compatible_additive`. Existing scalar definitions/interactions are unchanged. Temporal type codes and resolved local tuples are intentionally Fluxer-only, require explicit opt-in, and need documented scalar alternatives for Discord.
- **Risks and unknowns:** timezone databases evolve, browser/server tzdata may temporarily disagree, future civil-time rules can change, strict near-now validation can reject a value after client preview, and native pickers require real locale/assistive testing. A future independently owned account/community setting could make the lower-precedence fallbacks available, but command work does not create or broaden those settings. The representational bounds are interoperability limits, not traffic-derived product demand. Pinned timezone fixtures, resolution mismatch tests, clock-boundary tests, and staged capability rollout are required.

### Canonical unchanged-publication no-op — QAD-232

- **Question:** When a registration request is semantically equal to the published state, what equality, validation, concurrency, response, rate, history, and audit contract avoids useless revisions without weakening authorization or compatibility?
- **Options considered:** publish a new revision for every accepted request; deduplicate only identical bytes or idempotency keys; accept hash equality without exact comparison; ignore stale preconditions when the desired state matches; or perform full validation and return a semantic no-op only for exact canonical equality.
- **User-approved answer:** authenticate/authorize, charge the route, parse/normalize, validate all limits/current policy, resolve stable identities, and build the prospective complete state before comparison. Equality covers the active command/category identities, definition/schema/metadata/localizations, ordered presentation/input semantics, category registry/membership/order, developer availability, canonical targeting sets, and authority fields. JSON object/map order and equivalent omitted defaults are normalized; semantically ordered collections remain ordered. Publication IDs/revisions/timestamps are excluded.
- **Collision/transition safety:** use the canonical hash to find a likely match, then compare canonical bytes/objects. A new/reserved identity allocation, key mapping, dormant/reactivation transition, category lifecycle change, target change, pending authority consequence, or any different active state is not a no-op. Equality does not clear an existing pending approval or grant availability.
- **Response and side effects:** return each compatible endpoint's ordinary success status/body and current committed revisions plus additive `Fluxer-Publication-Status: unchanged`. Allocate no identity/revision/manifest, emit no publication/catalog/audit event, invalidate no cache, and create no durable no-op history. Store only bounded aggregate no-op metrics. A no-change draft remains intact and explicitly discardable. Repeating DELETE for an absent/dormant active resource preserves compatible unknown-resource behavior.
- **Concurrency:** an explicit stale expected manifest/draft/head version always returns its existing conflict even if the current state now happens to equal the requested result. Without an explicit precondition, a CAS loser may reread once and succeed as no-op only when the newly current canonical state exactly equals the already validated complete candidate; otherwise it returns manifest conflict. Fluxer never silently rebases a partial request or merges manifests.
- **Current and supplied evidence:** QAD-165/169 already require canonical definitions/hashes and immutable head CAS; QAD-171/172 define errors/rate charging; QAD-194 owns candidate cleanup. The current research still said every successful mutation creates a manifest. The PDF suggests skipping unchanged registrations but does not define semantic equality, authorization, status compatibility, or races.
- **Why this is sound:** semantic canonical comparison handles startup libraries that reorder JSON or omit defaults, while full validation and exact bytes prevent a hash/idempotency shortcut becoming an authorization or collision boundary. Preserving explicit preconditions protects concurrency intent; allowing only exact convergent races avoids pointless conflicts without merging.
- **Compatibility classification:** `compatible`. Existing compatible success objects/statuses and handler behavior remain unchanged; the response header is additive. No-op affects internal history/events only when observable resource state is already identical.
- **Risks and unknowns:** canonicalizer drift can produce false change/no-change, equality across category/target/authority state is security-sensitive, and over-labeling metrics could create cardinality/privacy problems. Shared canonical fixtures, hash-plus-byte comparison, race tests, and no-event/cache/audit assertions gate rollout.
- **Downstream contracts:** amend registration, persistence/publication, errors/rates, dashboard, audit, abuse/resource, compatibility, and C/M rollout contracts. SIM-C01/C03 must cover reordered/default-equivalent input, every near-match, explicit stale heads, convergent/divergent races, startup repetition, draft behavior, and absence of revision/event/history side effects.
- **Authority and currency:** the user approved the compact QAD-232 recommendation on 2026-07-20. The PDF remains non-authoritative under the recorded provenance limits.
